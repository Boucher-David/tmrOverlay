# Track Map Overlay Logic

Implementation files:

- `src/TmrOverlay.App/Overlays/TrackMap/TrackMapOverlayDefinition.cs`
- `src/TmrOverlay.App/Overlays/TrackMap/TrackMapForm.cs`
- `src/TmrOverlay.App/TrackMaps/TrackMapStore.cs`
- `src/TmrOverlay.App/TrackMaps/IbtTrackMapBuilder.cs`
- `src/TmrOverlay.Core/TrackMaps/TrackMapModels.cs`

The Track Map overlay is a transparent map-only surface. It draws the stored/generated track geometry when a map is available, otherwise it falls back to a simple circle. It does not draw title text, status text, panels, footers, borders, or other chrome.

Live car dots are placed by valid live `LapDistPct` only when the timing row is explicitly marked as having spatial progress, so the map only plots cars with usable position telemetry. The Track Map does not invent positions for missing live rows and does not use proximity/relative rows as a completeness signal. When session scoring data is available, marker labels and class colors prefer `LiveTelemetrySnapshot.Models.Scoring` for the matching `CarIdx`; live timing still supplies the lap-distance position. This keeps rendered dots in their real plotted location without compressing standings order when iRacing is only transmitting a partial car set.

The focused car is highlighted and can show its current scoring-backed `P<N>` position inside the dot. The player/team car is only highlighted when it is also the focused car. Other plotted timing rows with valid lap-distance progress are drawn as smaller class-colored dots. If a generated map exists, dots are interpolated along that map's `racingLine`; otherwise the same progress values are projected onto the circle fallback. Native and localhost renderers smooth marker progress between telemetry samples so dots move around the map without visible quarter-second stepping.

The track outline itself is always drawn at full opacity. The Track Map opacity setting is treated as map-fill opacity only: it changes the dark interior fill of the generated layout or circle fallback without dimming the line, pit lane, marker dots, sector highlights, or position label. This keeps the track readable when users want a lighter or darker internal map area.

Track-map sector status is part of live model v2 through `LiveTelemetrySnapshot.Models.TrackMap`. The normal map remains a continuous white line. When the focused/local driver completes a personal-best sector split for the session, only that sector segment is overpainted green. When the completed lap is a session-best lap, all sectors are overpainted purple; a personal-best lap with live session-best delta proving it is not the session best uses green. The full-lap highlight is intentionally temporary: after the next lap's first sector boundary is completed, the remaining map returns to white and only the current sector status remains highlighted.

Live sector timing accepts valid focused/local `LapDistPct` even when `LapCompleted`/`CarIdxLapCompleted` is unavailable or `-1`. A clean start/finish wrap is treated as a synthetic lap increment, so crossing the line can still seed the next sector run. Large discontinuities, such as active-reset jumps to a later point on the track, are not scored across the skipped distance; if the new progress is near a known sector boundary, the tracker seeds from that boundary and begins timing the following sector.

Track maps are not generated by the overlay itself. The overlay reads app-owned derived map documents through `TrackMapStore`. User-generated maps live under:

```text
%LOCALAPPDATA%\TmrOverlay\track-maps\user
```

Bundled maps, when present, are read from `Assets/TrackMaps` in the app output and are eligible even when IBT generation is disabled. User-generated maps are eligible only while the Track Map setting `track-map.build-from-telemetry` is enabled; otherwise the runtime path uses bundled maps first and falls back to the circle placeholder.

## Generation

After a successful IBT analysis, `TelemetryCaptureHostedService` can ask `IbtTrackMapBuilder` to derive a map from the selected `.ibt` file. User IBT generation is enabled by default from the Track Map settings tab through `track-map.build-from-telemetry`. Fresh installs use bundled app map JSON when available and otherwise keep the circle fallback until local generation produces a map. Locally generated user maps are read by the native overlay and localhost route only while that setting remains enabled. The settings tab explains that derived map geometry is generated locally from the user's telemetry-generated IBT files and stored on the user's machine. Source `.ibt` files are not copied into TMR storage by default.

The builder:

- reads `Lat`, `Lon`, `LapDistPct`, optional `LapDist`, lap number, speed, and pit-road state from the IBT payload;
- uses complete positive-numbered on-track laps only;
- falls back from invalid `LapCompleted` values to valid started-lap `Lap` values when grouping IBT samples;
- ignores pit-road and near-stationary racing-line samples;
- projects latitude/longitude to local meter coordinates;
- bins samples by `LapDistPct` and stores median `x/y` points;
- stores pit lane as a separate open geometry when enough pit-road samples exist;
- records confidence and quality metrics for complete laps, missing bins, closure, length delta, and repeatability;
- writes schema-v2 sector metadata from `SplitTimeInfo.Sectors` so renderers can map live model-v2 sector highlights onto generated geometry.

Generation is skipped when a complete map already exists for the normalized track/layout identity. A runtime-complete map means the document schema is supported, the generation version is current, confidence is at least `Medium`, and the generated racing line has enough points for its bin count. Runtime readers accept schema v1 maps for geometry compatibility, but newly generated and bundled map documents use schema v2 and include sector metadata. Better maps can still replace weaker maps; for example, a later `High` confidence map can replace a complete `Medium` map.

Track identity is layout-aware: the map key includes `TrackID` when present, normalized track name, normalized config/layout name, track length, and track version. This avoids treating alternate layouts as the same map when session info provides enough identity.

Diagnostics bundles include `metadata/track-maps.json`, a compact inventory of readable and invalid bundled/user map documents. It includes schema version, generation version, and sector count so support can distinguish stale geometry-only maps from schema-v2 sector-capable maps, and can separate a real map-rendering issue from an expected circle fallback when the current track identity has no matching bundled or user-generated map.

## Localhost

The localhost browser-source route for Track Map is:

```text
http://localhost:8765/overlays/track-map
```

The route is listed in the Track Map settings tab and does not depend on the native overlay being visible. The browser version uses the same map-only shape: transparent page, generated map or circle fallback, full-opacity track outline, map-fill opacity from settings, model-v2 sector highlights, focused-car `P<N>` text inside the dot, smoothed live dots, and the same wrap-aware marker progress smoothing as the native overlay. It fetches live telemetry from `/api/snapshot` on a 100 ms interval and map geometry/settings separately from `/api/track-map` so large map documents are not sent on every live update.

## Bundled Map Assets

`tools/TmrOverlay.TrackMapGenerator` is the batch path for turning the ignored `captures/IBT` corpus into derived JSON maps for `src/TmrOverlay.App/Assets/TrackMaps`. It writes derived geometry plus schema-v2 sector metadata; raw `.ibt` files stay ignored and must not be committed.

## Windows Validation

Track-map generation needs Windows/iRacing validation because the production app uses `irsdkSharp` and iRacing's own IBT telemetry writer. On a Windows machine with the .NET 8 SDK:

```powershell
dotnet restore .\tmrOverlay.sln
dotnet build .\tmrOverlay.sln --configuration Release --no-restore
```

For an isolated manual run:

```powershell
$env:TMR_Storage__AppDataRoot = "$env:TEMP\TmrOverlay-map-validate"
$env:TMR_TelemetryCapture__RawCaptureEnabled = "true"
$env:TMR_IbtAnalysis__Enabled = "true"
$env:TMR_IbtAnalysis__TelemetryLoggingEnabled = "true"
dotnet run --project .\src\TmrOverlay.App\TmrOverlay.App.csproj --configuration Release
```

In the settings window, open Track Map and leave `Build local maps from IBT telemetry` checked. Run iRacing long enough to record at least two complete clean laps for a usable map, preferably five or more for high confidence, then stop the session and let iRacing finish writing the `.ibt` file. The generated map should appear under:

```text
%TEMP%\TmrOverlay-map-validate\track-maps\user
```

Useful evidence to report back:

- the generated map JSON file name and its `quality` block;
- the matching `track_map_generated`, `track_map_generation_rejected`, or `track_map_generation_skipped` event from `logs/events/*.jsonl`;
- the capture's `ibt-analysis/status.json`;
- whether `http://localhost:8765/overlays/track-map` shows a real track shape or the circle fallback.

To test bundled-map generation without launching the app:

```powershell
dotnet run --project .\tools\TmrOverlay.TrackMapGenerator\TmrOverlay.TrackMapGenerator.csproj -- --source "$env:USERPROFILE\Documents\iRacing\telemetry" --output .\src\TmrOverlay.App\Assets\TrackMaps --min-confidence Medium --dry-run
```

Remove `--dry-run` only after reviewing the confidence output and expected asset count.

The `Build local maps from IBT telemetry` checkbox does not scan existing telemetry by itself. It controls future automatic post-session generation and whether user-generated maps are eligible at runtime. Use `tools/TmrOverlay.TrackMapGenerator` for direct bundled-asset generation and QA against an existing telemetry folder.
