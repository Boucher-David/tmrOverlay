# Track Map Overlay Logic

Implementation files:

- `src/TmrOverlay.App/Overlays/TrackMap/TrackMapOverlayDefinition.cs`
- `src/TmrOverlay.App/Overlays/TrackMap/TrackMapForm.cs`
- `src/TmrOverlay.App/TrackMaps/TrackMapStore.cs`
- `src/TmrOverlay.App/TrackMaps/IbtTrackMapBuilder.cs`
- `src/TmrOverlay.Core/TrackMaps/TrackMapModels.cs`

The Track Map overlay is a transparent map-only surface. It draws the stored/generated track geometry when a map is available, otherwise it falls back to a simple circle. It does not draw title text, status text, panels, footers, borders, or other chrome.

Live car dots are placed by `LapDistPct`. The focused/player car is highlighted, and other timing rows with valid lap-distance progress are drawn as smaller class-colored dots. If a generated map exists, dots are interpolated along that map's `racingLine`; otherwise the same progress values are projected onto the circle fallback.

Track maps are not generated by the overlay itself. The overlay reads app-owned derived map documents through `TrackMapStore`. User-generated maps live under:

```text
%LOCALAPPDATA%\TmrOverlay\track-maps\user
```

Bundled maps, when present, are read from `Assets/TrackMaps` in the app output and are eligible even when IBT generation is disabled. User-generated maps are eligible only while the Track Map setting `track-map.build-from-telemetry` is enabled; otherwise the runtime path uses bundled maps first and falls back to the circle placeholder.

## Generation

After a successful IBT analysis, `TelemetryCaptureHostedService` can ask `IbtTrackMapBuilder` to derive a map from the selected `.ibt` file. User IBT generation is enabled by default from the Track Map settings tab through `track-map.build-from-telemetry`. Fresh installs use bundled app map JSON when available and otherwise keep the circle fallback until local generation produces a map. Locally generated user maps are read by the native overlay and localhost route only while that setting remains enabled. The settings tab explains that derived map geometry is generated locally from the user's telemetry-generated IBT files and stored on the user's machine. Source `.ibt` files are not copied into TMR storage by default.

The builder:

- reads `Lat`, `Lon`, `LapDistPct`, optional `LapDist`, lap number, speed, and pit-road state from the IBT payload;
- uses complete positive-numbered on-track laps only;
- ignores pit-road and near-stationary racing-line samples;
- projects latitude/longitude to local meter coordinates;
- bins samples by `LapDistPct` and stores median `x/y` points;
- stores pit lane as a separate open geometry when enough pit-road samples exist;
- records confidence and quality metrics for complete laps, missing bins, closure, length delta, and repeatability.

Generation is skipped when a complete map already exists for the normalized track/layout identity. A runtime-complete map means the document schema/generation versions are current, confidence is at least `Medium`, and the generated racing line has enough points for its bin count. Better maps can still replace weaker maps; for example, a later `High` confidence map can replace a complete `Medium` map.

Track identity is layout-aware: the map key includes `TrackID` when present, normalized track name, normalized config/layout name, track length, and track version. This avoids treating alternate layouts as the same map when session info provides enough identity.

## Localhost

The localhost browser-source route for Track Map is:

```text
http://localhost:8765/overlays/track-map
```

The route is listed in the Track Map settings tab and does not depend on the native overlay being visible. The browser version uses the same map-only shape: transparent page, generated map or circle fallback, and live dots. It fetches live telemetry from `/api/snapshot` and map geometry separately from `/api/track-map` so large map documents are not sent on every live update.

## Bundled Map Assets

`tools/TmrOverlay.TrackMapGenerator` is the batch path for turning the ignored `captures/IBT` corpus into derived JSON maps for `src/TmrOverlay.App/Assets/TrackMaps`. It writes derived geometry only; raw `.ibt` files stay ignored and must not be committed.

## Windows Validation

Track-map generation needs Windows/iRacing validation because the production app uses `irsdkSharp` and iRacing's own IBT telemetry writer. On a Windows machine with the .NET 8 SDK:

```powershell
dotnet restore .\tmrOverlay.sln
dotnet build .\tmrOverlay.sln --configuration Release --no-restore
```

For an isolated manual run:

```powershell
$env:TMR_Storage__AppDataRoot = "$env:TEMP\TmrOverlay-map-validate"
$env:TMR_LocalhostOverlays__Enabled = "true"
$env:TMR_TelemetryCapture__RawCaptureEnabled = "true"
$env:TMR_IbtAnalysis__Enabled = "true"
$env:TMR_IbtAnalysis__TelemetryLoggingEnabled = "true"
dotnet run --project .\src\TmrOverlay.App\TmrOverlay.App.csproj --configuration Release
```

In the settings window, open Track Map and leave `Build local maps from IBT telemetry` checked. Run iRacing long enough to record at least two complete clean laps for a usable map, preferably five or more for high confidence, then stop the session and let iRacing finish writing the `.ibt` file. The generated map should appear under:

```text
%TEMP%\TmrOverlay-map-validate\track-maps\user
```

Useful evidence to report back:

- the generated map JSON file name and its `quality` block;
- the matching `track_map_generated`, `track_map_generation_rejected`, or `track_map_generation_skipped` event from `logs/events/*.jsonl`;
- the capture's `ibt-analysis/status.json`;
- whether `http://localhost:8765/overlays/track-map` shows a real track shape or the circle fallback.

To test bundled-map generation without launching the app:

```powershell
dotnet run --project .\tools\TmrOverlay.TrackMapGenerator\TmrOverlay.TrackMapGenerator.csproj -- --source "$env:USERPROFILE\Documents\iRacing\telemetry" --output .\src\TmrOverlay.App\Assets\TrackMaps --min-confidence Medium --dry-run
```

Remove `--dry-run` only after reviewing the confidence output and expected asset count.

The `Build local maps from IBT telemetry` checkbox does not scan existing telemetry by itself. It controls future automatic post-session generation and whether user-generated maps are eligible at runtime. Use `tools/TmrOverlay.TrackMapGenerator` for direct bundled-asset generation and QA against an existing telemetry folder.
